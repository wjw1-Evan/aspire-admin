
# 后端开发总览

> 本规范结合 AGENTS.md 最高准则，统一后端开发标准，所有开发、评审、文档、提交均须严格遵循。

## 1. 数据访问与多租户

- **禁止**直接注入或操作 `IMongoCollection<T>`、`IMongoDatabase`，所有数据操作**必须**通过注入 `DbContext` 完成。
- 业务服务层通过 `_context.Set<T>()` 增删查改，自动享受多租户（CompanyId）过滤、软删除、审计字段（CreatedAt/UpdatedAt/CreatedBy/UpdatedBy/IsDeleted/DeletedAt/DeletedBy）自动填充。
- 禁止在查询中手动拼接 CompanyId，全部由全局过滤器自动处理。

## 2. 控制器分层与 DTO 下沉

- 控制器层**只负责**路由、参数校验、权限注解 `[RequireMenu]`、调用服务层、返回统一响应。
- **禁止**在控制器直接注入/操作 DbContext、IMongoCollection、IMongoDatabase。
- 请求/响应 DTO 必须定义在 Models/ 目录，禁止在控制器内定义。
- 复杂业务、批量处理、关联查询等逻辑必须全部下沉到服务层封装。

## 3. 权限与鉴权

- 所有敏感操作**必须**使用 `[RequireMenu("menu-action")]` 注解，严禁使用过时的 `HasPermission()`、`RequirePermission()` 等方法。
- 除 userId 外，任何企业上下文（companyId/菜单/角色）**禁止**直接从 JWT claim 读取，统一通过注入 `ITenantContext` 获取。

## 4. 统一响应与中间件

- 所有 HTTP 接口返回数据必须包装为 `ApiResponse<T>`，默认 camelCase，忽略 null 字段。
- 严格遵循中间件注册顺序，禁止随意调整（详见 03-中间件与响应.md）。

## 5. 实时通信与 SSE

- 后端**禁止**再向前端使用 SignalR 发送实时数据，全部改为 SSE（Server-Sent Events），连接管理器为 `IChatSseConnectionManager`。
- SSE 响应需跳过统一响应中间件格式化。

## 6. 命名、提交与文档

- 所有代码注释、README、Git 提交信息**必须**使用简体中文。
- Commit 信息遵循约定式提交（如 `feat: 添加xxx`，`fix: 修复xxx`，`docs: 更新xxx`）。
- DTO、模型、服务、控制器命名需统一风格，禁止 using 别名（详见 07-类型命名规范.md）。

## 7. 前端开发标准

- 前端页面开发标准参考**密码本模块**：[前端页面开发标准](../frontend-admin/06-页面开发标准.md)
- 前端必须使用 `ApiResponse<T>` 包装 API 响应
- 类型定义统一在 `@/types/api-response.ts`

## 8. 变更建议与文档维护

- 各子文档如有原则重复，优先合并至本总览，其他文档引用本规范避免重复。
- 所有规范需结合实际代码持续修订，发现实现与文档不符时，优先以 AGENTS.md 和本总览为准。

---

## 9. 卫星服务（Storage / SystemMonitor）

- **Platform.SystemMonitor** 使用独立 HTTP 宿主；认证方式为 **JWT**（与 ApiService 同源配置）及 **`X-Internal-Service-Key`** 服务间调用，详见 [微服务拆分开发指南](../../guides/MICROSERVICE-GUIDE.md)。
- **文件存储**：GridFS 文件存储已集成到 `Platform.ApiService` 内部，通过 `IStorageClient` 接口调用。
- 业务层 **DbContext 红线** 针对 `Platform.ApiService` 等业务服务；Storage 内 GridFS 访问不经过 `PlatformDbContext`，但仍须遵守多租户与权限在 **ApiService** 侧的约束，避免绕过业务 API 暴露敏感对象。
