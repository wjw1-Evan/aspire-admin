# 流程设计与审批功能说明

## 1. 功能定位

本说明聚焦 Aspire 管理系统中的“流程设计 + 审批执行”功能模块。侧重描述业务行为和流程逻辑，免去接口细节。目标是提供完整功能理解、设计规范和核心执行路径，便于产品、开发与测试团队协同。

### 1.1 核心价值

- 可视化流程设计（开始/审批/条件/结束）
- 流程实例化（流程定义快照、变量、文档关联）
- 多种审批策略（任意/会签/顺序）
- 统一审批动作（通过、拒绝、退回、转办）
- 审批历史与状态追踪
- 多租户隔离与权限校验

## 2. 关键概念

- **流程定义**：图结构 + 节点/边，包含审批配置、条件分支、表单绑定。
- **流程实例**：单份文档或业务对象在流程中的运行记录，包含当前节点、审批人、审批记录、变量、快照。
- **审批节点**：Form or approval 节点，配置审批人、类型、操作许可。
- **动作**：`Approve`/`Reject`/`Return`/`Delegate`。
- **状态**：`Pending`、`Running`、`Waiting`、`Completed`、`Rejected`、`Cancelled`。

## 3. 流程设计规则（引擎校验）

流程校验服务确保流程定义合法：

1. 1 个开始节点、>=1 个结束节点。
2. 节点唯一，无自环或重复边。
3. 从开始节点可达全部节点。
4. 开始节点必须有出边，结束节点必须有入边，其它节点不能孤立。
5. 审批节点须配置审批规则 (`approvers` 非空)。
6. 条件节点须有条件边或默认边。
7. 支持节点类型：`start/end/approval/condition`。

## 4. 审批策略

审批节点依据 `ApprovalType` 不同:

- `Any`（或签）：任一审批人通过节点完成。
- `All`（会签）：全部审批人都通过节点完成。
- `Sequential`（顺序）：按审批人顺序逐一执行，前一个通过后激活下一人，完全通过后完成。

## 5. 审批动作

- `Approve`：记录审批结果，按策略判断节点是否完成，完成则继续流转。
- `Reject`：直接将流程实例置 `Rejected` 并结束。
- `Delegate`：可将当前审批任务转给他人，更新当前审批人列表，继续当前节点。
- `Return`：退回到流程内指定节点，支持流程回滚。

## 6. 审批人规则

支持多种方式定义审批人：

- 静态用户（UserId）
- 角色（RoleId）
- 部门（DepartmentId）
- 表单字段提取（FormFieldKey）

同时支持抄送 (`ccRules`) 逻辑，便于通知与跟踪。

## 7. 流程状态与执行控制

- 流程在实例化时记录流程定义快照、表单定义快照，防止定义变更影响历史流程。
- 核心状态由引擎维护：当前 `CurrentNodeId`、`CurrentApproverIds`、`ActiveApprovals`。
- 流程节点完成后自动走下一节点；遇结束节点标记 `Completed`。
- 处理并发与防越权：仅当前审批权人员可执行；顺序审批仅当前轮到人员可审批。

## 8. 审批直观特性

- 审批记录包含 `审批人/动作/意见/时间/顺序`，形成可审计轨迹。
- 退回和转办支持补录说明，记录操作链路。
- 策略支持配置节点 `allowReject`、`allowReturn`、`allowDelegate`，灵活开关。

## 9. 通知机制（功能补充）

配合工作流引擎，触发通知事件：

- 工作流启动、待审批、审批完成、审批拒绝、转办、流程结束。
- 通知对象：审批人、发起人、转办人等。

## 10. 场景举例

1. 报销单提交：发起人发起后进入第一审批节点，按部门角色审批；通过后流转到财务复核；最终完成。
2. 采购流程：条件节点根据金额分支执行不同审批链，支持顺序审批（逐级审批）。
3. 设备借用：单人审批 node `Any`，任一委员通过即完成；拒绝直接终止。
4. 文档退回：审批人可退回到“填写节点”重新修改后继续。

## 11. 国际成熟系统参考与最佳实践

本项目功能已具备核心能力，以下借鉴国际成熟流程与审批平台（如 BPMN/Camunda/Flowable/Activiti/IBM BPM/Oracle BPM）补充要点：

- 采用标准化流程建模概念：显式 `start` / `end` / `gateway`(条件、并行、排他) / `task` / `subprocess`。
- 明确事件模型：支持信号事件、定时器事件、消息事件，提高流程可观测与可恢复性。
- 设计异常路径：`error`/`cancel`/`escalation`，在节点异常时可回滚或触发补偿逻辑。
- 设定 SLA 监控：流程和节点级别超时阈值，自动告警与自动触发下一步策略（超时审批自动通过/拒绝/通知）。
- 版本管理与回滚策略：流程定义版本历史可追溯，实例绑定定义版本，运行中定义变更不影响当前实例，历史审计一致。
- 灵活并行与循环支持：并行分支 `AND` 审批、分支合并、循环执行。（当前流程模型可在未来支持）
- 精细权限控制：基于 RBAC/ABAC 的审批操作权限，可设“仅自身审批/仅角色审批/仅指定用户”与“跨租户隔离”。
- 表单与流程解耦：流程只关注节点与审批规则，具体数据模式使用可扩展表单定义，支持动态字段、模板与关联数据。
- 审计与追踪：流程追踪ID、审批链路、操作上下文写入链路追踪系统 (例如 Jaeger/Zipkin)；日志级别与审计日志分离。
- 可测试化：流程场景支持“模拟器/沙箱执行”，与 CI 集成自动化测试（拒绝、回退、分支、并行、恢复）

## 12. 国际标准落地建议（进一步优化）

- 逐步补全 `条件网关` 与 `并行网关` 逻辑，实现更复杂业务路径。
- 引入 `Subprocess`（子流程），支持流程复用与流程嵌套。
- 支持 `signal/message` 触发流程与外部系统交互，如审批结果回写 SAP/ERP。
- 提供 `流程可视化审计回放` 功能，还原运行轨迹与状态变化。
- 在审批规则中加入“动态优先级”、“超时补偿规则”、“异常补偿任务”。
- 增加“审批人选缺失补全规则”，自动找候选人/代办人。
